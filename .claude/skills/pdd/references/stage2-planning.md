## 优先级矩阵

### Value vs Cost 矩阵

```
         高价值                    低价值
高成本 ┌─────────────┬─────────────┐
       │ 谨慎评估    │ 不推荐      │
       │ 高ROI才做   │ 除非必要    │
       ├─────────────┼─────────────┤
低成本 │ MVP优先     │ 后续迭代    │
       │ 立即执行    │ 有空再做    │
       └─────────────┴─────────────┘
```

### 优先级定义

| 优先级 | 定义 | 示例 |
|--------|------|------|
| P0 | MVP必需，阻塞发布 | 核心功能、关键修复 |
| P1 | 重要功能，影响用户体验 | 性能优化、增强功能 |
| P2 | 增强功能，提升体验 | UI改进、辅助功能 |
| P3 | 可选功能，锦上添花 | 高级设置、个性化 |

## 任务分解

### 分解原则

1. **独立性**: 每个任务应尽量独立，减少依赖
2. **原子性**: 一个任务对应一个明确可交付成果
3. **可估算**: 工时应在0.5-8小时范围内
4. **可验证**: 有明确的完成标准

### 任务模板

```markdown
### [TASK-ID] 任务名称
- **类型**: 后端/前端/移动端/架构/测试/DevOps
- **优先级**: P0/P1/P2/P3
- **预计工时**: Xh
- **依赖**: [TASK-XXX]
- **描述**: 详细描述
- **验收标准**:
  - [ ] 标准1
  - [ ] 标准2
```

## 工作量估算

### 估算技术

#### 1. 类比估算法
- 参考类似历史任务
- 调整复杂度差异

#### 2. 三点估算法
- 乐观时间 (O)
- 悲观时间 (P)
- 最可能时间 (M)
- 估算 = (O + 4M + P) / 6

#### 3. 规划扑克
- 团队成员独立估算
- 讨论差异
- 达成共识

### 常见任务参考工时

| 任务类型 | 简单 | 中等 | 复杂 |
|----------|------|------|------|
| API端点 | 1-2h | 2-4h | 4-8h |
| UI组件 | 1-2h | 2-4h | 4-8h |
| 数据库迁移 | 0.5-1h | 1-2h | 2-4h |
| 单元测试 | 0.5-1h | 1-2h | 2-4h |
| 文档编写 | 0.5h | 1h | 2h |

### 依赖管理

#### 依赖类型
- **硬依赖**: 必须等待前置任务完成
- **软依赖**: 可以并行，但建议顺序执行
- **外部依赖**: 依赖外部团队或服务

#### 依赖图示例
```
[TASK-001] API设计 ──► [TASK-002] 后端实现 ──► [TASK-004] 集成测试
                              │
                              └──► [TASK-003] 前端实现
```

## 常见问题

### Q: 任务太大怎么办？
A: 分解为更小的子任务：
1. 按功能点分解
2. 按技术层次分解
3. 按开发阶段分解

### Q: 如何处理不确定的估算？
A:
1. 增加缓冲时间 (20-30%)
2. 创建探针任务验证假设
3. 使用范围估算而非精确值

### Q: 多个任务可以并行吗？
A: 可以，但需注意：
- 确认无硬依赖
- 考虑代码冲突风险
- 保持沟通同步
