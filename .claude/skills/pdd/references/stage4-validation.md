## 业务价值评估

### Value Score 公式

```
Value Score = (User Value × 0.4) + (Business Value × 0.4) + (Technical Value × 0.2)
```

### 评分标准

| 维度 | 评分因素 | 1-3分 | 4-6分 | 7-10分 |
|------|----------|-------|-------|--------|
| 用户价值 | 影响范围、痛点程度 | 少数用户、轻微痛点 | 部分用户、明显痛点 | 大量用户、严重痛点 |
| 商业价值 | 收入、成本、竞争 | 微小影响 | 明显影响 | 重大影响 |
| 技术价值 | 架构、可维护性 | 增加技术债务 | 中性或轻微改进 | 显著改进架构 |

### 决策矩阵

| Value Score | 建议行动 |
|-------------|----------|
| > 8 | 继续投资，考虑扩展功能 |
| 5-8 | 优化改进，收集更多反馈 |
| < 5 | 重新评估，考虑调整或下线 |

## 完成或迭代

### 价值达成 - 完成

条件:
- Value Score > 8
- 所有P0需求通过
- 关键KPI达成

行动:
1. 移动PRD到 `specs/completed/`
2. 创建发布说明
3. 准备上线

```bash
mv specs/active/[module-name] specs/completed/
```

### 价值未达成 - 迭代

条件:
- Value Score 5-8
- 部分需求未通过
- 需要优化

行动:
1. 分析根本原因
2. 更新PRD
3. 返回Stage 2

### 价值不足 - 重新评估

条件:
- Value Score < 5
- 核心需求未通过
- ROI不达标

行动:
1. 深入分析原因
2. 考虑调整方案
3. 或考虑终止

## 常见问题

### Q: Value Score低于预期怎么办？
A:
1. 分析各维度得分
2. 找出主要短板
3. 制定改进计划
4. 评估是否值得继续投入

### Q: 用户反馈与预期不符怎么办？
A:
1. 收集更多反馈数据
2. 分析用户行为差异
3. 考虑用户群体匹配
4. 必要时调整产品定位

### Q: 验证报告应该多详细？
A:
- 完整模式: 包含所有细节
- 标准模式: 聚焦关键发现
- 快速模式: 简化但包含决策依据
