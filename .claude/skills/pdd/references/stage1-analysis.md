# Stage 1: 产品分析与需求定义

## 概述

**负责人**: 产品经理

**触发条件**: 任何用户输入、市场反馈或业务请求

**目标**: 深入理解需求，创建完整的PRD文档

---

## 工作流程

### 1. 分析用户需求

#### 1.1 理解痛点
- 用户当前面临什么问题？
- 问题发生的频率如何？
- 问题的影响范围有多大？
- 现有解决方案的不足之处？

#### 1.2 评估业务价值
- 这个功能能带来多少收入或节约多少成本？
- 对用户留存/获取有什么影响？
- 与竞争对手相比的差异化？

#### 1.3 评估产品契合度
- 是否符合产品战略方向？
- 是否与现有功能互补？
- 是否会引入技术债务？

### 2. 定义成功指标

#### SMART原则
- **S**pecific: 具体明确
- **M**easurable: 可量化衡量
- **A**chievable: 可实现
- **R**elevant: 与目标相关
- **T**ime-bound: 有时限

#### 常见KPI示例
| 类型 | 指标示例 |
|------|----------|
| 用户增长 | DAU/MAU, 新注册用户数 |
| 用户参与 | 使用时长, 功能使用率 |
| 用户满意 | NPS, 用户评分, 反馈 |
| 商业指标 | 收入, 转化率, ARPU |
| 技术指标 | 响应时间, 错误率 |

## PRD创建

### 文件位置
```bash
mkdir -p specs/active/[module-name]
cp specs/templates/requirement-template.md specs/active/[module-name]/requirement.md
```

### 必要内容清单

#### 用户故事格式
```
作为 [用户角色]
我希望 [完成的目标]
以便于 [获得的价值]
```

#### 验收标准格式
```
给定 [前置条件]
当 [用户执行操作] 时
则 [预期结果]
```

#### 商业价值定义
| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 用户价值 | 40% | 影响用户数 × 痛点程度 |
| 商业价值 | 40% | 收入影响 + 成本节约 |
| 技术价值 | 20% | 架构改进 + 可维护性 |

## 常见问题

### Q: 如何判断MVP范围是否合理？
A: MVP应包含解决核心痛点的最小功能集。使用以下测试：
1. 没有这个功能，用户能完成主要任务吗？
2. 这个功能是否解决了80%的核心问题？
3. 后续迭代是否能基于MVP扩展？

### Q: 什么时候需要创建完整PRD vs 简化PRD？
A:
- **完整PRD**: 新功能模块、架构变更、影响多个团队
- **简化PRD**: 小改动、配置更新、UI微调
- **Quick模式**: 可使用简化的PRD模板
